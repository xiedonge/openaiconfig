# 配置管理网站需求文档

## 1. 项目概述

本项目是一个部署在服务器上的私有配置管理后台，当前目标是统一管理并切换 `codex` 与 `openclaw` 的运行配置。

网站的核心价值不是单纯做一套增删改查，而是：

- 保存多条配置档案
- 让管理员登录后管理这些配置
- 在点击“启用”时，把选中的配置实时写入服务器上的真实配置文件
- 在每次修改真实配置文件前自动生成备份
- 支持在网站中查看备份列表并执行还原
- 让 `codex` 和 `openclaw` 后续都能复用同一套后台与数据结构

## 2. 当前确定范围

### 2.1 用户模型

- 只有一个管理员账号
- 必须登录后才能进入后台
- V1 不做注册
- V1 不做多用户
- V1 不做角色权限系统

### 2.2 支持的应用

- V1 支持 `codex`
- 本轮需求同时纳入 `openclaw`
- 后续可以继续扩展更多应用类型

### 2.3 配置项字段

每条配置记录包含以下业务字段：

- `名称`
- `URL`
- `API Key`

系统字段由后台维护：

- `应用类型`
- `是否启用`
- `创建时间`
- `更新时间`
- `最近应用时间`
- `最近应用状态`
- `最近应用结果`

### 2.4 需要支持的操作

- 登录
- 查看配置列表
- 新增配置
- 编辑配置
- 删除配置
- 启用配置
- 查看备份列表
- 还原备份

## 3. 产品目标

### 3.1 核心目标

- 提供一个你自己可用的私有配置管理后台
- 安全地管理多个 `codex` / `openclaw` 配置
- 允许在页面中快速切换当前启用配置
- 将启用动作与真实配置文件写入绑定
- 让 API Key 默认脱敏显示

### 3.2 非目标

- 不做公开官网首页
- 不做访客访问
- 不做多人协作
- 不做复杂审批流
- 不做多服务器分发

## 4. 页面范围

### 4.1 登录页

路由：`/login`

页面元素：

- 用户名输入框
- 密码输入框
- 登录按钮
- 错误提示区域

行为要求：

- 支持回车提交
- 密码默认隐藏
- 登录成功后进入配置列表页
- 已登录用户访问登录页时自动跳转到配置页

### 4.2 配置列表页

路由：`/configs`

这是后台主页面。

页面元素：

- 页面标题
- 退出登录按钮
- 备份列表入口
- 应用类型筛选
- 新增配置按钮
- 配置表格或卡片列表

列表展示字段：

- 名称
- 应用类型
- URL
- API Key
- 是否启用
- 最近应用状态
- 更新时间
- 操作列

操作列需要支持：

- 编辑
- 删除
- 启用
- 显示 API Key
- 隐藏 API Key

API Key 展示规则：

- 默认脱敏显示
- 支持点击后显示完整值
- 支持再次隐藏
- 刷新页面后默认恢复为脱敏状态

### 4.4 备份列表页

路由：`/backups`

这是用于查看配置文件备份与执行还原的页面。

页面元素：

- 页面标题
- 应用类型筛选
- 备份来源筛选
- 备份列表
- 还原按钮

列表展示字段：

- 备份编号
- 应用类型
- 关联配置
- 触发来源
- 包含文件
- 创建时间
- 最近还原状态
- 操作列

操作列需要支持：

- 查看备份文件明细
- 执行还原

还原行为要求：

- 还原前需要二次确认
- 还原成功后需要立即刷新列表状态
- 如果还原的是 `openclaw` 备份，还原完成后还需自动执行 `openclaw gateway restart`
- 如果还原关联的配置记录仍然存在，则该配置应恢复为当前启用配置
- 如果还原关联的配置记录已被删除，则还原成功后该应用类型下不标记任何启用记录

### 4.3 新增/编辑配置

建议实现方式：

- V1 先做弹窗表单

表单字段：

- 应用类型
- 名称
- URL
- API Key

校验规则：

- 应用类型必填
- 名称必填
- URL 必填且格式合法
- API Key 必填

保存规则：

- 新增时创建一条新记录
- 编辑时更新对应记录
- 保存不等于启用

## 5. 主要使用流程

### 5.1 登录流程

1. 用户打开网站
2. 未登录时跳转到 `/login`
3. 输入管理员用户名和密码
4. 登录成功后进入 `/configs`
5. 登录失败则停留在登录页并显示错误信息

### 5.2 日常操作流程

1. 管理员登录
2. 查看某个应用类型下的配置列表
3. 新增、编辑、删除或启用配置
4. 点击“启用”后，系统先对将被修改的真实配置文件创建备份
5. 备份成功后，系统写入对应应用的真实配置文件
6. 如果应用类型为 `openclaw`，系统还需要自动执行 `openclaw gateway restart`
7. 所有必需步骤成功后，数据库中该配置才变为启用状态
8. 任一步骤失败时，数据库启用状态保持不变，并反馈失败原因

### 5.3 备份还原流程

1. 管理员进入备份列表页
2. 按应用类型或触发来源筛选备份
3. 查看某条备份包含的文件
4. 点击“还原”
5. 系统先对当前真实配置文件再次创建一份还原前备份
6. 系统用选中的备份覆盖目标配置文件
7. 如果应用类型为 `openclaw`，系统自动执行 `openclaw gateway restart`
8. 还原成功后更新备份状态与当前启用配置状态
9. 还原失败时记录失败原因

## 6. 核心业务规则

### 6.1 单应用单启用规则

- 同一个 `appType` 下，同一时间只能有一条启用配置
- `codex` 和 `openclaw` 各自独立维护自己的启用记录

### 6.2 启用成功判定

点击“启用”后，只有当目标配置文件全部成功写入时，这条配置才算启用成功。

如果任一步骤失败：

- 数据库中的启用状态不切换
- 当前已启用的旧配置保持不变
- 记录失败原因

分应用判定规则：

- `codex`：`~/.codex/config.toml` 与 `~/.codex/auth.json` 都写入成功，才算启用成功
- `openclaw`：`~/.openclaw/openclaw.json` 写入成功且 `openclaw gateway restart` 执行成功，才算启用成功

如果 `openclaw` 写入成功但重启命令失败：

- 此次启用判定为失败
- 数据库中的启用状态不切换
- 系统应自动尝试使用本次写入前刚生成的备份回滚配置文件
- 回滚完成后应再次执行 `openclaw gateway restart`
- 回滚结果与重启结果都要记录下来

### 6.3 备份与还原规则

- 任何会修改真实配置文件的动作，都必须先生成备份
- 备份按“备份集”管理，一次动作可以包含一个或多个备份文件
- `codex` 的一次备份集应包含：
  - `config.toml`
  - `auth.json`
- `openclaw` 的一次备份集应包含：
  - `openclaw.json`
- 网站中必须能列出已有备份集及其包含的备份文件
- 用户可以在网站中选择某个备份集执行还原
- 还原动作本身也必须先生成一份新的“还原前备份”
- `codex` 备份还原后可立即生效，不需要额外命令
- `openclaw` 备份还原后必须自动执行 `openclaw gateway restart`
- 如果还原关联的配置记录仍然存在，则该记录恢复为当前启用配置
- 如果还原关联的配置记录已被删除，则还原成功后该应用类型下不标记任何启用配置

### 6.4 删除规则

- 删除为直接删除
- V1 不弹二次确认
- 即使当前记录处于启用状态，也允许删除
- 删除当前启用记录后：
  - 数据库中该记录被移除
  - 该应用类型下暂时没有任何“启用中”的记录
  - 已经写入服务器配置文件的内容不会被自动回滚

### 6.5 敏感信息规则

- API Key 默认脱敏显示
- 服务端日志中不得打印完整 API Key
- 未认证接口不得返回完整 API Key
- 前端只有在已登录状态下才能查看完整值

### 6.6 登录态规则

- `/configs` 及相关接口都必须鉴权
- 未登录访问页面时跳转到登录页
- 未登录访问接口时返回 `401`
- 退出登录后销毁会话

## 7. 应用配置映射规则

## 7.1 Codex 映射规则

目标目录：

- `~/.codex`

目标文件：

- `~/.codex/config.toml`
- `~/.codex/auth.json`

字段映射：

- 网站中的 `URL` 写入 `config.toml` 顶层字段 `base_url`
- 网站中的 `API Key` 写入 `auth.json` 顶层字段 `OPENAI_API_KEY`

已知结构：

- 当前目录中的 `config.toml` 存在顶层 `base_url`
- 当前目录中的 `auth.json` 存在顶层 `OPENAI_API_KEY`

生效行为：

- `codex` 配置更新后可立即生效
- 不需要重启服务

## 7.2 Openclaw 映射规则

目标目录：

- `~/.openclaw`

目标文件：

- `~/.openclaw/openclaw.json`

本轮先按当前目录示例文件中的 provider key 处理，实际使用的节点为：

- `models.providers.custom-goood-my.baseUrl`
- `models.providers.custom-goood-my.apiKey`

说明：

- 你消息里给出的路径一处写成了 `custom-good-my`，另一处写成了 `custom-goood-my`
- 当前目录中的示例文件实际出现的是 `custom-goood-my`
- V1 文档先按示例文件中的真实 key `custom-goood-my` 设计

写入规则：

- 网站中的 `URL` 写入 `models.providers.custom-goood-my.baseUrl`
- 网站中的 `API Key` 写入 `models.providers.custom-goood-my.apiKey`

当前示例文件现状：

- 当前目录中的 `openclaw.json` 不是合法 JSON
- 在 `channels` 与 `gateway` 之间缺少逗号
- 这说明实现时不能假设文件永远合法，必须把 JSON 解析失败视为“启用失败”

生效行为：

- `openclaw` 配置写入成功后，系统必须自动执行 `openclaw gateway restart`
- 只有“写入成功 + 重启命令成功”，此次启用才算成功
- 如果重启命令失败，系统必须自动尝试用本次写入前备份回滚 `openclaw.json`
- 回滚后系统应再次执行 `openclaw gateway restart`
- 如果回滚仍失败，需要记录高优先级错误并提示人工处理

## 8. 数据模型设计

建议建立 `configs` 表。

建议字段：

- `id`
- `appType`
- `name`
- `url`
- `apiKey`
- `isActive`
- `createdAt`
- `updatedAt`
- `lastAppliedAt`
- `lastApplyStatus`
- `lastApplyMessage`

字段说明：

- `id`：主键
- `appType`：应用类型，当前值包括 `codex`、`openclaw`
- `name`：配置名称
- `url`：目标 URL
- `apiKey`：目标 API Key
- `isActive`：是否为当前启用配置
- `createdAt`：创建时间
- `updatedAt`：更新时间
- `lastAppliedAt`：最近一次写文件时间
- `lastApplyStatus`：最近一次写文件状态，建议值为 `never`、`success`、`failed`
- `lastApplyMessage`：最近一次写文件结果描述

约束建议：

- 同一 `appType` 下只允许一条记录为 `isActive = true`
- `appType` 建索引
- `updatedAt` 建索引

### 8.2 备份数据模型

建议建立 `backup_sets` 与 `backup_files` 两张表。

`backup_sets` 建议字段：

- `id`
- `appType`
- `triggerType`
- `relatedConfigId`
- `createdAt`
- `lastRestoreAt`
- `lastRestoreStatus`
- `lastRestoreMessage`
- `postActionStatus`
- `postActionMessage`

字段说明：

- `appType`：应用类型，如 `codex`、`openclaw`
- `triggerType`：备份触发来源，建议值如 `activate`、`restore`
- `relatedConfigId`：关联的配置记录 ID，可为空
- `createdAt`：备份集创建时间
- `lastRestoreAt`：最近一次还原时间
- `lastRestoreStatus`：最近一次还原结果
- `lastRestoreMessage`：最近一次还原描述
- `postActionStatus`：还原或启用后附加动作结果，例如 `openclaw gateway restart`
- `postActionMessage`：附加动作说明

`backup_files` 建议字段：

- `id`
- `backupSetId`
- `sourcePath`
- `backupPath`
- `fileName`
- `createdAt`

字段说明：

- `backupSetId`：所属备份集 ID
- `sourcePath`：原始文件路径
- `backupPath`：备份文件路径
- `fileName`：备份文件名
- `createdAt`：创建时间

## 9. 登录与认证设计

V1 推荐方案：

- 不单独建管理员表
- 管理员用户名和密码哈希放在环境变量

建议环境变量：

- `ADMIN_USERNAME`
- `ADMIN_PASSWORD_HASH`
- `SESSION_SECRET`

安全要求：

- 密码只保存哈希，不保存明文
- 会话 Cookie 使用 `HttpOnly`
- 生产环境启用 `Secure`
- 建议设置 `SameSite=Lax` 或更严格

## 10. 后端功能模块

## 10.1 认证模块

职责：

- 校验用户名和密码
- 创建登录态
- 保护管理页面和接口
- 支持退出登录

## 10.2 配置 CRUD 模块

职责：

- 查询配置列表
- 新增配置
- 编辑配置
- 删除配置

要求：

- 所有接口都必须登录后访问
- 服务端必须做字段校验
- 校验错误要返回明确提示

## 10.3 启用模块

职责：

- 根据配置记录找到对应应用适配器
- 先创建备份集
- 写入目标配置文件
- 执行应用特定的后置动作
- 在写入成功后切换数据库启用状态
- 记录最近应用时间和结果

要求：

- 启用必须是独立接口或服务端动作
- 文件写入与数据库状态切换应当作为一个逻辑事务处理
- 文件写入失败时不得切换启用状态
- 如果后置动作失败，系统必须记录失败并按规则尝试回滚

## 10.4 应用适配器层

为了后续支持更多应用，不能把文件写入逻辑散落在各个接口中。

建议为每个 `appType` 建立独立适配器。

每个适配器负责：

- 定义所需字段
- 定义目标文件路径
- 定义字段映射
- 定义文件解析和写入逻辑
- 定义错误处理逻辑

初始适配器：

- `codex` 适配器
- `openclaw` 适配器

## 10.5 备份与还原模块

职责：

- 创建备份集与备份文件记录
- 查询备份列表与备份文件明细
- 执行备份还原
- 在还原后执行应用相关后置动作
- 更新还原结果与关联配置状态

要求：

- 所有备份与还原操作都必须登录后访问
- 还原前必须先创建一份当前状态备份
- `openclaw` 还原后必须执行 `openclaw gateway restart`
- 还原成功后，需要同步修正数据库中的启用配置状态

## 11. 文件写入要求

### 11.1 运行环境要求

网站部署的服务进程必须与目标配置目录处于同一台服务器，并拥有对应写权限。

需要可写的目标文件：

- `~/.codex/config.toml`
- `~/.codex/auth.json`
- `~/.openclaw/openclaw.json`

### 11.2 路径解析要求

不要把用户家目录写死成绝对路径。

应在运行时解析当前进程用户的 home 目录，然后拼出：

- `~/.codex/config.toml`
- `~/.codex/auth.json`
- `~/.openclaw/openclaw.json`

### 11.3 写入策略

对于 `codex`：

- `config.toml` 需要安全解析 TOML
- 只更新 `base_url`
- 保留其他无关配置

- `auth.json` 需要安全解析 JSON
- 只更新 `OPENAI_API_KEY`
- 保留其他无关字段

对于 `openclaw`：

- `openclaw.json` 需要安全解析 JSON
- 只更新 `models.providers.custom-goood-my.baseUrl`
- 只更新 `models.providers.custom-goood-my.apiKey`
- 保留其他无关字段
- 写入完成后必须执行 `openclaw gateway restart`

### 11.4 安全写入流程

建议采用以下流程：

1. 读取原文件
2. 解析原文件
3. 构造更新后的内容
4. 创建备份集并保存原文件副本
5. 记录备份元数据
6. 写入临时文件
7. 再用临时文件替换原文件

### 11.5 备份策略

V1 要求：

- 每次修改真实配置文件前都必须创建带时间戳的备份集
- 备份集必须能在网站中被列出
- 备份文件必须可用于后续还原
- 一次动作涉及多个目标文件时，应归入同一个备份集

推荐备份目录：

- `~/.codex/backups/<backupSetId>/config.toml`
- `~/.codex/backups/<backupSetId>/auth.json`
- `~/.openclaw/backups/<backupSetId>/openclaw.json`

后续可扩展：

- 自动清理旧备份
- 按时间或数量保留策略清理

### 11.6 还原流程

建议采用以下流程：

1. 读取选中的备份集信息
2. 校验备份文件存在且可读
3. 对当前真实配置文件创建一份新的“还原前备份”
4. 将备份文件内容覆盖回目标路径
5. 如果应用类型为 `openclaw`，执行 `openclaw gateway restart`
6. 根据还原结果更新数据库中的启用配置状态
7. 记录最近还原时间、状态与结果说明

### 11.7 错误处理

以下情况都必须判定为启用失败：

- 目标文件不存在
- 目标文件不可读
- 目标文件解析失败
- 目标文件不可写
- 权限不足
- 目标 JSON 节点不存在
- `openclaw gateway restart` 执行失败
- 回滚失败

错误信息要求：

- 能帮助排查问题
- 不泄露完整 API Key

## 12. 接口设计建议

### 12.1 登录相关接口

- `POST /api/login`
- `POST /api/logout`
- `GET /api/session`

### 12.2 配置相关接口

- `GET /api/configs?appType=codex`
- `GET /api/configs?appType=openclaw`
- `POST /api/configs`
- `PUT /api/configs/:id`
- `DELETE /api/configs/:id`
- `POST /api/configs/:id/activate`

### 12.3 备份相关接口

- `GET /api/backups`
- `GET /api/backups?appType=codex`
- `GET /api/backups?appType=openclaw`
- `GET /api/backups/:id`
- `POST /api/backups/:id/restore`

接口原则：

- 返回明确的成功或失败状态
- 返回清晰的人类可读错误信息
- 不在日志中打印完整密钥

## 13. 前端交互建议

### 13.1 列表排序

建议默认排序：

- 当前启用记录优先
- 其次按更新时间倒序

### 13.2 状态展示

- 启用中的记录要有明显标识
- 最近应用失败的记录要显示失败状态
- 最近应用结果可用标签、提示框或行内说明展示

### 13.3 API Key 脱敏格式

建议展示方式：

- 保留前 4 位
- 保留后 4 位
- 中间使用 `*` 替代

示例：

- `sk-12ab********89xy`

### 13.4 错误反馈

- 表单校验错误显示在字段附近
- 启用失败时给出 toast 或行内错误提示
- 登录失败停留在登录页并显示原因

### 13.5 备份列表交互

- 备份列表默认按创建时间倒序
- 能清楚区分“激活前备份”和“还原前备份”
- 能展示每个备份集包含的备份文件
- 还原成功后要显示成功提示
- 还原失败后要显示失败原因

## 14. 技术方案建议

推荐实现：

- 前端：`Next.js`
- 后端：`Next.js Route Handlers` 或服务端动作
- 数据库：`SQLite`
- 登录：基于 Session 的单管理员登录
- 部署：Node.js 服务部署在目标服务器上

选择原因：

- 结构简单
- 适合单人自用后台
- 易于直接访问本机配置文件
- 后续扩展 `openclaw` 成本低

## 15. 部署要求

部署前提：

- 网站服务和目标配置文件在同一台服务器
- 服务进程拥有 `~/.codex` 与 `~/.openclaw` 的读写权限
- 服务进程所在环境必须可以执行 `openclaw gateway restart`
- 数据库文件位于持久化目录
- 管理员账号相关环境变量正确配置
- 生产环境使用 HTTPS

## 16. 验收标准

满足以下条件即可视为 V1 达标：

1. 未登录用户不能访问配置后台。
2. 管理员可以成功登录和退出。
3. 可以新增 `codex` 配置。
4. 可以新增 `openclaw` 配置。
5. 可以编辑任意已有配置。
6. 可以直接删除任意配置。
7. 列表中 API Key 默认脱敏显示。
8. 列表中可以显示和隐藏完整 API Key。
9. 同一个应用类型下只能有一条启用配置。
10. 启用 `codex` 配置时，会更新 `~/.codex/config.toml` 的 `base_url`。
11. 启用 `codex` 配置时，会更新 `~/.codex/auth.json` 的 `OPENAI_API_KEY`。
12. 启用 `openclaw` 配置时，会更新 `~/.openclaw/openclaw.json` 中 `models.providers.custom-goood-my.baseUrl`。
13. 启用 `openclaw` 配置时，会更新 `~/.openclaw/openclaw.json` 中 `models.providers.custom-goood-my.apiKey`。
14. 启用 `openclaw` 配置后，系统会自动执行 `openclaw gateway restart`。
15. 只有全部目标文件写入成功，且必须的后置动作执行成功，对应配置才会变成启用状态。
16. 任一写入失败时，数据库启用状态不切换。
17. 任一后置动作失败时，数据库启用状态不切换，并记录失败原因。
18. 每次修改真实配置文件前，系统都会生成备份集。
19. 网站中可以列出备份集及其包含的备份文件。
20. 网站中可以执行备份还原。
21. 还原 `openclaw` 备份后，系统会自动执行 `openclaw gateway restart`。
22. 还原成功后，系统会同步更新对应应用的启用配置状态。
23. 系统会记录最近应用时间、最近应用状态、最近还原状态。

## 17. 当前已知风险与待确认项

### 17.1 Openclaw 示例文件问题

当前目录中的 `openclaw.json` 示例文件存在 JSON 语法错误：

- `channels` 对象后缺少逗号
- 导致该文件无法直接作为标准 JSON 解析

这不影响需求定义，但会影响真实实现验证。

### 17.2 Openclaw 节点命名差异

目前消息描述与示例文件之间存在以下差异：

- 消息中出现过 `custom-good-my`
- 示例文件实际为 `custom-goood-my`

当前文档先按示例文件中的真实 key `custom-goood-my` 设计。

### 17.3 Openclaw 命令执行环境

当前需求已经明确：修改 `openclaw` 配置后需要执行 `openclaw gateway restart`。

实现前仍需保证：

- 服务器环境中可直接执行 `openclaw` 命令
- 服务进程拥有执行该命令的权限
- 命令的超时与返回码处理策略要在实现阶段定清楚

### 17.4 自动回滚边界

对于 `openclaw`：

- 如果“写入成功但重启失败”，系统应自动尝试回滚
- 如果“回滚后再次重启”仍失败，则需要记录为高优先级故障
- 此时系统要提示人工介入处理
